# Story 6.4: Execute coupled-pair strategy in paper/testnet mode

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Investment Manager,
I want the Trading Co. to execute strategy in paper/testnet mode behind a clean interface,
so that simulated positions/P&L flow to the ledger without any real venue or capital (FR-20, NFR-7).

## Acceptance Criteria

1. **Given** a strategy execution in paper/testnet mode (no real CEX, no real capital), **when** it runs, **then** it produces ledger entries tagged to the executing entity (`TRADING_CO`), and simulated P&L accrues to the VCC via its ownership chain (the consolidated group NAV) and is visible in the group view.
2. **Given** the execution path, **when** it is implemented, **then** it sits behind a clean interface seam so it can be re-implemented in Rust/Go post-P0 without caller changes (NFR-7), and it is small-scale only.

### Derived acceptance criteria (binding — from FR-16 / addendum §threshold-only, D1/D1a, NFR-2/NFR-4/NFR-9)

3. **Threshold-only, NEVER a clock.** A rebalance/reset fires **only** when a losing leg breaches the floor `f`; it is **never** triggered by elapsed time, an interval, or a scheduler. The executor exposes **no** timer/clock surface. A tick that stays within the barrier is a strict **no-op** (no lifecycle transition, no on-chain call, no ledger entry).
4. **Refuse-if-absent floor.** The floor parameters `m`/`g` (`MODEL_FLOOR_M`/`MODEL_FLOOR_G`) are read from `@rose/config` `loadConfig` — absence is a **refusal** (`ConfigRefusalError`), never a permissive default and never `0` (NFR-4, §11.2). The per-pair leverage `L` is read **from the pair row** (never hard-coded).
5. **Reset crystallizes P&L (D1/D1a), fail-closed, at the on-chain commit point.** On a breach the executor drives the lifecycle transition (`ACTIVE → REBALANCING`), then — gated by the default-deny capital-flow authorization **pre-write** (fail-closed, NFR-4) — drives the paired-burn dual-write (Epic 5.4); at the confirmed on-chain commit point (synthetic `PairBurned` in paper) it posts **exactly one balanced journal entry** crystallizing the realized P&L tagged to `TRADING_CO` (the losing holder bears the loss; the realized gain is withdrawable) and re-bases the pair symmetrically (re-anchor P₀ ← tick price, `V_A = V_B = K/2`, `K` conserved), returning the pair to `ACTIVE`. Idempotent under re-delivery (NFR-9); all amounts integer `bigint` (NFR-2).

### Scope boundary (P0, this story only — PAPER/LOCAL)

- **Network perimeter (binding, strict).** "Paper/testnet" = **paper/local**: local Postgres, the on-chain paired burn exercised through a **mock EIP-1193 transport / injected seam wallet** (the same precedent as 5.4 burn + 6.2/6.3), the `PairBurned` confirmation supplied as a **synthetic confirmed `ChainEvent`**, and price **ticks provided/simulated** (NO real CEX, NO real price feed, NO real capital). **Refuse-if-absent on every network secret and on the parked floor params. Create NO `.env`, NO secret, NO placeholder RPC/address/key.** The REAL Sepolia broadcast + the REAL price-feed/oracle wiring + the live `watchPairEvents` reset cadence are **ops-deferred** (recorded in `deferred-work.md`, story-6.4 section), exactly as 5.4/6.2/6.3 deferred their real broadcasts.

- **IN SCOPE:**
  - (a) **The strategy execution layer in the existing `@rose/rose-note` package** (`prod/packages/rose-note`) — the architecture places paper execution (FR-20) here alongside subscription/redemption ([architecture.md §Project Structure line 319; §Requirements-to-Structure line 357 — `Rose Note lifecycle & paper execution | FR-11, FR-20, FR-21 | prod/packages/rose-note`]). It COMPOSES the already-built seams (it authors NO new ledger/chain/authorization primitive) behind a clean injected port (NFR-7):
    1. **`StrategyExecutor` port** (the NFR-7 seam — a clean interface so the executor can be re-implemented in Rust/Go post-P0 without caller changes) with a concrete TS impl `makeStrategyExecutor(deps)`. `deps` are INJECTED (no connection opened, no key held, no env read here): `{ db, burn: BurnPairDualWrite, pairAddress, authorize: PairAuthorizationGate, topology: StrategyResetTopology, paymentAsset, floor: { modelFloorM, modelFloorG } }` (the floor `m`/`g` are resolved by the composition root from `loadConfig`, which refuses-if-absent).
    2. **`onTick(tick): Promise<StrategyTickOutcome>`** — the **threshold-only** trigger. Reads the pair (`getCoupledPair`), derives the floor threshold `floorUnits` from the parked `m`/`g` + the per-pair `L` + the pool `K` (`f = m·L·g`; `floorUnits = ⌊(K/2)·f⌋`, exact decimal→units arithmetic via `@rose/shared`/BigInt — **no float, NFR-2**), and compares each leg's **tick-supplied marked value** against it. **No breach ⇒ `{ action: 'none' }`, nothing written.** Breach (a losing leg's marked value ≤ `floorUnits`) **and pair `ACTIVE` ⇒** transition `ACTIVE → REBALANCING`, then drive `burn.start({ idempotencyKey: resetIdempotencyKey, …, authorize })` (authorize PRE-submit, fail-closed) → SUBMITTED (tx hash, **no ledger entry yet** — no optimistic success). Idempotent on `resetIdempotencyKey` (a reused key whose recorded intent mismatches the request ⇒ `StrategyResetIdempotencyConflictError`, 409).
    3. **`confirmReset(event: PairBurnedEvent): Promise<StrategyResetView | null>`** — THE COMMIT POINT (driven by the synthetic confirmed event in paper / `watchPairEvents` in live): build the reset `BurnLedgerPlan` and delegate to `burn.confirmFromBurnedEvent(event, plan)` → ONE balanced journal entry posted at the commit point (the retired reset-delta token quantity from the confirmed on-chain `amount`, D3/NFR-9, **plus the VALUE postings crystallizing the realized P&L tagged to `TRADING_CO`** — the losing holder bears the loss, the realized gain withdrawable, issuer-neutral & balanced per (asset, scale)). On `applied`, re-base the pair (`applyCoupledPairReset`: re-anchor P₀ ← tick price, `V_A = V_B = splitInTwo(K)`) and transition `REBALANCING → ACTIVE`. **NEVER throws into the (fire-and-forget) watcher** (a malformed/divergent event ⇒ caught, logged, `null`; the row stays SUBMITTED for reconcile 5.6). Idempotent (re-delivery ⇒ no-op, no double re-base).
    4. **`getReset(id): Promise<StrategyResetView | null>`** — derive `pending`/`confirmed`/`failed` from the outbox `PAIR_BURN` row keyed by the reset idempotency key (the shared outbox row-kind guard, mirroring 6.2/6.3 `viewFromRow`; a non-reset row ⇒ `null`).
    5. Structured INFO/WARN logging at the decision points (no-breach no-op, breach + transition, submit, commit, re-base) with pairId / losingLeg / floorUnits / resetIdempotencyKey / txHash context (CLAUDE.md §11).
  - (b) **`buildStrategyResetBurnPlan(input): BurnLedgerPlan`** (`prod/packages/rose-note/src/strategy-plan.ts`) — derives the `@rose/chain` 5.4 `BurnLedgerPlan` for a reset from a caller-supplied `StrategyResetTopology` (the established `PairLedgerPlan` caller-supplied-facts trust boundary): the **token-quantity legs** retire the reset-delta quantity (both holder accounts ASSET-classified, each supply contra non-ASSET — mirror the 5.4/6.3 burn topology so the retired quantity reconciles to on-chain `totalSupply`, NFR-9), and the **VALUE postings** crystallize the realized P&L: the losing holder's loss is settled and the realized gain made withdrawable (issuer-neutral, equal amounts), **plus a `TRADING_CO` tag posting pair** (the executing entity records the realized trading P&L — e.g. DEBIT `DEPLOYED_CAPITAL` / CREDIT `FEE_INCOME`, same payment asset, equal amount) so the entry is **tagged to `TRADING_CO`** and the P&L **accrues to the group NAV** (AC-1). Every (asset, scale) group balances; the plan satisfies the 5.4 `assertPairPlanAccountsDisjoint` guard. `InvalidStrategyResetError` on a non-positive/float reset amount (NFR-2). All `bigint`.
  - (c) **One additive `@rose/ledger` primitive — `applyCoupledPairReset`** (`prod/packages/ledger/src/repositories/coupled-pairs.ts`): a NEW, **additive** function (it does NOT change any existing function, signature, schema, or migration) that, in a row-locking transaction, re-anchors a pair (`anchor_price ← newAnchorPrice`) and re-bases its legs (`long_leg_value`/`short_leg_value ← supplied symmetric values, `K` unchanged`), advancing `updated_at`. **Guarded:** it refuses unless the pair is in the reset window (`REBALANCING` or `PARTIAL`) — a `CoupledPairResetStateError`; it validates inputs with the same frozen-type guards as `createCoupledPair` (positive decimal anchor, max-scale, non-negative integer leg values) and asserts `V_A + V_B = K` (conservation). It writes only EXISTING columns ⇒ **no new migration** (migrations stay at 7). This is the persisted symmetric re-base the executor applies at the commit point.
  - (d) **`@rose/api` strategy endpoints** branching onto the 6.1 boundary, symmetric to the 6.2/6.3 write endpoints: `POST /coupled-pairs/:id/strategy/ticks` (submit a tick; Zod-validated body; money/marks as integer STRINGS, NFR-2; returns the `StrategyTickOutcome`) and `GET /strategy/resets/:id` (read the pending/confirmed reset status). The route calls an INJECTED `StrategyExecutor` port (`ApiDeps.strategy?`); when not wired (paper mode not composed) the write path is a typed **503** refusal (`STRATEGY_SERVICE_UNAVAILABLE`, refuse-if-absent), never an opaque 500.
  - (e) Extend the 6.1 structured-error registry (`@rose/api` `errors.ts`) with the new typed refusal classes this path raises: `StrategyResetIdempotencyConflictError` ⇒ **409**, `CoupledPairResetStateError` ⇒ **409**, `InvalidStrategyResetError` ⇒ **422**, `CoupledPairNotFoundError` (REUSE from `@rose/ledger`) ⇒ **404**, `ConfigRefusalError` (REUSE from `@rose/config`) ⇒ **503** (parked-param refusal — refuse-if-absent, never a generic 500).   - (f) **The reset loop proven LOCALLY:** a `@rose/rose-note` test drives (seed an ACTIVE pair embedded in a note + a seeded minted position →) a **within-barrier tick ⇒ strict no-op** (no transition, no outbox row, no entry — the threshold-only / never-clock proof, incl. repeated identical ticks and ticks "later in time" doing nothing) → a **floor-breach tick ⇒** transition to `REBALANCING` + paired burn submitted (pending, no entry) → synthetic confirmed `PairBurned` ⇒ `confirmReset` posts EXACTLY ONE balanced journal entry **tagged to a `TRADING_CO` account**, re-bases the pair (`V_A = V_B = K/2`, new anchor) and returns it to `ACTIVE`; `buildGroupView` reflects the realized P&L in `TRADING_CO`'s balances and the group NAV (AC-1), and the ledger circulating token quantity reconciles to the (synthetic) post-burn on-chain supply (NFR-9). Three adversarial lenses ran against the Story-6.4 diff. The **Acceptance Auditor** (diff + story + epics/addendum) returned PASS on AC-1 (the commit-point entry is tagged to `TRADING_CO` `DEPLOYED_CAPITAL`/`FEE_INCOME`, balanced, and the realized P&L shows in `TRADING_CO`'s balances + the consolidated group NAV, with token quantity reconciling to the post-burn on-chain supply, NFR-9), AC-2 (clean `StrategyExecutor` NFR-7 seam, injected port, no `@rose/chain`/`viem` edge leaks into `@rose/api`), AC-3 (threshold-only — no `Date`/timer/scheduler in the decision path; a within-barrier tick writes nothing; proven incl. repeated/"later" ticks), AC-4 (parked `m`/`g` refuse-if-absent via `@rose/config`; `L` read from the row; exact BigInt/decimal arithmetic), and the Epic-7 BOUNDARY (no `/throwaway` import, no `V_A=(K/2)(1±L·r)` reimplementation, marks opaque). The **Blind Hunter** (diff only) and the **Edge-Case Hunter** (live local Postgres probing) jointly surfaced three correctness hazards, all variants of "the pair is stranded in `REBALANCING` after the irreversible burn has posted," which were FIXED:

1. **(High, FIXED) Unvalidated `tick.price` strands the pair after the burn.** `onTick` stashed the raw price and only validated it at `confirmReset` → `applyCoupledPairReset` (which throws on a negative / >8-decimal anchor) AFTER the burn entry was already committed; the throw was swallowed and re-delivery was a `noop`, so the pair never re-based/returned to `ACTIVE`. FIX: validate `price` UP-FRONT in `onTick` (positive decimal, ≤8 fractional digits → `InvalidStrategyResetError`, 422) — a refusal now leaves the pair `ACTIVE`. +3 regression tests.
2. **(High, FIXED) Submit failure / second authorize after the transition stranded `REBALANCING`.** `onTick` transitioned `ACTIVE→REBALANCING` BEFORE `burn.start`, so a submit failure (or a flipped second authorize evaluation) left a dangling `REBALANCING` with no recoverable path. FIX: reorder to submit the burn FIRST (the 5.4 dual-write runs the SAME fail-closed default-deny gate pre-submit and throws before recording), then transition only on success — a refusal/failure leaves the pair `ACTIVE`. Single authorize evaluation.
3. **(High, FIXED) Re-base/transition failure after the burn had no recovery.** The re-base was guarded by `outcome.status === 'applied'` only, so a re-delivery (`noop`) skipped it forever. FIX: widen the guard to run on `applied` OR `noop` while the pair is still `REBALANCING`, making recovery idempotent (a later delivery finishes a partially-completed reset). An `anomaly` (divergence/zero) still leaves the row SUBMITTED for reconcile 5.6.

Additionally **(Med, FIXED)** a terminally FAILED/COMPENSATED existing reset row is now reported truthfully (`action: 'none', reason: 'reset-failed:<status>'`) instead of a misleading `reset-started` with a null tx hash.
