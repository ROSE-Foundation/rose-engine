# Story 5.4: Burn the coupled token package on redemption with matching ledger entries

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Investment Manager,
I want redeeming a Note to burn the whole coupled token package (both legs) on-chain and record the matching balanced ledger entry,
so that retirement is real on-chain, atomic, and reconcilable to the books (FR-21).

## Acceptance Criteria

1. **Given** a redemption, **when** the burn executes, **then** the whole package (both legs, L-Token + S-Token) is burned at equal amount via the custom `CoupledPair.burnPair(lFrom, sFrom, amount)` on Sepolia (atomic — both-or-neither), and the ledger records the matching balanced entry in ONE balanced journal entry (with FR-13) capturing the burned token **quantity** (and the redemption **value** notional). The dual-write flows through the Story-5.2 outbox/saga: the on-chain `burnPair` tx is the **commit point** — the balanced journal entry is posted **only after** the `PairBurned` confirmation (`ChainEvent`), never at intent or at submission (NFR-3).
2. **Given** the redeemed position, **when** reconciliation reads on-chain quantities, **then** post-redemption on-chain supply and ledger token quantities reconcile (NFR-9): the burned on-chain quantity matches the retired ledger quantity, and a single-leg burn is impossible. Replaying the same `PairBurned` (watcher re-delivery / reorg re-scan) does NOT double the ledger retirement (idempotent commit point, from 5.2).

### Scope boundary (P0, this story only)

- IN SCOPE: the concrete **paired-burn dual-write orchestration** wired onto the 5.2 `OutboxSaga`, MIRRORING the proven 5.3 paired-mint pattern — (a) a typed on-chain `burnPair` write seam (`submitBurnPair`, the saga `submit` port) calling the epic-4 `CoupledPair.burnPair(lFrom, sFrom, amount)` via the 5.1 `getWalletClient`; `encodeBurnPairCall` is the pure, network-free calldata seam; (b) a concrete `LedgerEffect` (`makeBurnPairLedgerEffect`) that, at the commit point, posts ONE balanced journal entry linked to the coupled pair (FR-13 `recordJournalEntry`) capturing the burned **token quantity** (taken from the confirmed on-chain `PairBurned.amount` — chain = source of truth, D3 / NFR-9) **and** the redemption **value** notional, fail-closed behind the authorization seam `postTransfer` consults; (c) the burn orchestration `BurnPairDualWrite` (record intent `PAIR_BURN` → submit → confirm-from-event) and the `PairBurned → confirm` wiring. All proven **LOCALLY** (real local Postgres for the ledger effect, mock EIP-1193 transport / in-memory fakes + synthetic `PairBurnedEvent`s for the orchestration + write encoding — NOT against real Sepolia, no wallet key).
- REUSE — factor, do not duplicate: the burn module shares the established invariant helpers/types with the 5.3 mint module where the semantics are identical (amount validation `assertPairAmount`/`InvalidPairAmountError`, the plan-disjointness guard, the divergence/plan error shapes, the `PairAuthorizationGate`/`PairAuthorizationDecision`, the `MAX_UINT256`/`HEX_TX_HASH` constants, the `PairLegAccounts`/`PairValuePlan` shapes). Extract the shared pieces into a small sibling module (e.g. `prod/packages/chain/src/pair-dual-write/shared.ts`) rather than copy-paste, OR keep the 5.3 exports as the canonical home and import them — whichever produces the smaller, clearer diff WITHOUT breaking the 5.3 public surface / re-exports. The burn-specific direction of the quantity postings is the only material difference (burn RETIRES supply: the holder leg is CREDITED and the supply contra DEBITED — the exact inverse of mint).
- OUT OF SCOPE (later stories — do NOT implement): the consolidated **group view** (5.5), the full ledger↔chain **reconcile-and-correct** loop + finality/reorg cadence (5.6), the Model-A principal-clamp-at-burn business rule beyond what the epic-4 contract already enforces on-chain (4.4 clamps principal at burn at the contract layer; the ledger effect records the confirmed on-chain burned quantity — it does not re-derive the clamp). This story delivers BURN/redemption + the balanced ledger recording only.
- OUT OF SCOPE (ops, deferred): the REAL Sepolia broadcast/confirmation of `burnPair` (no `SEPOLIA_RPC_URL`, no signing key, no confirmation-depth tuning against a live chain). The write code (`submitBurnPair`) is code-complete and proven locally; the actual broadcast awaits secrets provided out-of-band. Record it in `deferred-work.md` (story-5.4 ops section). **No real secret is ever created.**

### P0 interpretation (documented, not invented scope)

- **"Matching balanced ledger entries" + "postTransfer-governed":** mirrors the 5.3 P0 interpretation. `postTransfer` records a single-asset, exactly-two-posting VALUE transfer and cannot, by itself, express the combined quantity(TOKEN)+value(notional) entry AC-1 mandates. FR-13's primitive (`recordJournalEntry` with `coupled_pair_id`) is what records a coupled event as ONE balanced entry. Therefore 5.4 records ONE combined balanced entry via `recordJournalEntry`, and honors "postTransfer-governed" by consulting the SAME substitutable default-deny authorization decision `postTransfer` consults (injected `PairAuthorizationGate`) BEFORE any write — fail-closed (NFR-4). This keeps the chokepoint's governance while satisfying the single-entry + FR-13 requirement, and keeps `@rose/chain` decoupled from `@rose/authorization` (no new edge/cycle). A future composition layer (Epic 6) injects the real `postTransfer` provider into the gate.
- **Account topology is caller-supplied:** like `postTransfer` and `issueCoupledPair`, the concrete L/S holder + supply-contra accounts and the value postings are supplied by the caller via `BurnLedgerPlan`. 5.4 does not model a fixed account layout (leg→token-account linkage is an Epic-6 concern); it guarantees the invariants (one balanced entry, quantity == on-chain amount, both legs always present, supply retired, fail-closed).
