# Story 6.1: Expose the typed REST API boundary (Fastify + Zod + OpenAPI)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a build engineer,
I want a typed Fastify REST API with Zod validation and an OpenAPI document,
so that surfaces and any audit consumers have an explicit, documented contract and authorization refusals surface cleanly (FR-14 foundation).

## Acceptance Criteria

1. **Given** the `api` package, **when** a request is handled, **then** request/response bodies are Zod-validated, money is serialized as decimal strings (never JS `number`), and an OpenAPI document is generated (derived from the same Zod schemas — single source of the I/O types).
2. **Given** an authorization refusal or domain rejection, **when** the API responds, **then** it returns the structured error `{ error: { code, message, details? } }` with the correct status (403 authorization refusal, 422 domain rule rejection, 409 invariant/idempotency conflict, 400 validation) **and** the `code`/`message` are specific enough for the surface to name the refusing rule to the user (UX-DR5) — refusals are never collapsed into a generic error.

### Scope boundary (P0, this story only)

- IN SCOPE: a new PROD package `@rose/api` (the architecture home for the Fastify REST boundary, `prod/packages/api`) delivering the TYPED API SURFACE that Stories 6.2→6.6 branch onto:
  (a) `buildApp(deps)` — a Fastify factory wired with the **Zod type provider** (`fastify-type-provider-zod`): request/response schemas are Zod, validation + serialization run through Zod, and `@fastify/swagger` derives the **OpenAPI 3.x** document from those SAME schemas (single source of I/O types). Dependencies (the ledger `RoseDb`, optional injected `ChainSupplySnapshot`) are INJECTED — the app opens no DB connection and no chain connection itself (injected-port precedent, LOCAL/in-process testable);
  (b) base READ-ONLY endpoints over already-available state (Epics 1–5): `GET /health`, `GET /group-view` (the consolidated group view from `@rose/reconcile`, money as decimal strings), `GET /coupled-pairs/:id` (read a coupled pair), `GET /rose-notes/:id` (read a Rose Note), `GET /openapi.json` (the generated OpenAPI document);
  (c) the typed structured-error contract `{ error: { code, message, details? } }` and a single error mapper that translates domain/authorization errors to the correct HTTP status — the headline being the authorization `TransferRefusedError` split (`DENY` → **403**, `REFUSE` → **422**, named-rule `code`/`message`, UX-DR5), plus 409 (invariant/idempotency conflict), 400 (Zod validation), 404 (not found), 503 (config refuse-if-absent), 500 (unexpected, non-leaking). This mapper is the boundary 6.2→6.6 write paths surface their refusals through.
- REUSE — factor, do not duplicate: `@rose/reconcile` `buildGroupView`/`serializeGroupView`/`groupViewToJson` (the consolidated read model + its no-`bigint`/no-float JSON); `@rose/ledger` `getCoupledPair`/`getRoseNote`/`RoseDb` (the read repos); `@rose/shared` `toDecimalString`/`assertNotFloat` style for any local money serialization (smallest-unit `bigint` → string). The four-fixed-entity / five-fixed-account-type model and the existing serializer style are REUSED, not re-modelled. `zod` is already a workspace dep (`@rose/config`/`@rose/chain`/`@rose/rule-spec`).
- OUT OF SCOPE (later stories — do NOT implement): the live subscription flow (6.2), live redemption (6.3), paper/testnet strategy execution (6.4), and the React/Vite UI surfaces — Covenant Console (6.5) / exchange + subscriber surfaces (6.6). NO write endpoints, NO mint/burn driving, NO chain calls. This story exposes the typed boundary + base read endpoints + OpenAPI; it lays what those stories branch onto, nothing more.
- OUT OF SCOPE (ops, deferred): serving the real consolidated group view against a LIVE Sepolia supply (the `ChainSupplySnapshot` is injected; the real chain read is the 5.5/5.6 ops seam already recorded). Binding the API to a real network listen port / process is an ops concern — the app is exercised purely via Fastify `inject` (in-process). **No real secret is ever created. No `.env`. No placeholder address/RPC.** Record the ops items in `deferred-work.md` (story-6.1 section).

### P0 interpretation (documented, not invented scope)

- **Status-code mapping for the structured error.** The AC names 403/422/409/400. The single documented interpretation: `TransferRefusedError` `DENY` → 403 (authorization refusal — default-deny, no rule allowed it), `REFUSE` → 422 (a rule explicitly refused — a domain rule rejection); domain validation/rejection errors → 422; double-entry / lifecycle / outbox invariant + idempotency conflicts → 409; Zod request/response validation → 400; absent resource → 404; refuse-if-absent config faults → 503 (server not configured — this also resolves the story-3.4 deferred concern that config/parse faults must NOT read as an opaque 500 at the REST boundary); anything unmapped → 500 with a generic, non-leaking message. The mapper is NAME-keyed so the boundary stays decoupled from importing every domain package (it reads `error.name`, and `TransferRefusedError.effect` for the 403/422 split). Revisable as the write endpoints (6.2→6.6) add their own error classes.
- **Coupled-pair magnitudes are smallest-unit integer strings (no fabricated scale).** K/V_A/V_B cross the wire as raw integer strings exactly as the `@rose/reconcile` group view surfaces them — the `coupled_pairs` row does not carry a per-leg decimal scale, so the boundary does not invent one. anchor/leverage/floor cross as their stored decimal strings.
