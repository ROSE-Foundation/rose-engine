# Story 5.1: Connect to Sepolia via typed viem clients and event watchers

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a build engineer,
I want typed viem clients and chain-event watchers for the deployed contracts,
so that the `chain` package can read on-chain state and observe token events (NFR-9 foundation).

## Acceptance Criteria

1. **Given** the `chain` package configured with the Sepolia contract addresses, **when** I read on-chain token balances and total supply, **then** the viem clients return typed results with ABI inference.
2. **Given** mint/burn/transfer events on-chain, **when** the viem watchers run, **then** confirmed events are surfaced as typed, decoded event objects ready for the `outbox_events`/reconcile pipeline (consumed by 5.2+), and the `chain` package is the only module talking to Sepolia.
3. **Given** the `chain` package started with `SEPOLIA_RPC_URL` and/or the contract addresses absent, **when** it attempts to build its clients, **then** it **refuses to start** with a clear error naming the missing key(s) — no placeholder RPC, no default address, no secret embedded (NFR-4, §11.2, refuse-if-absent).

### Scope boundary (P0, this story only)

- IN SCOPE: typed viem public client (reads) + a wallet-client factory seam (writes, used by 5.3/5.4), typed event watchers for the epic-4 contract events (at minimum `PairMinted`/`PairBurned`, plus the `RoseToken` `Transfer`), refuse-if-absent chain config, ABI-derived TypeScript types, all proven **LOCALLY** (mock EIP-1193 transport / in-process — NOT against real Sepolia).
- OUT OF SCOPE (later stories — do NOT implement): outbox/saga dual-write & idempotency (5.2), paired mint orchestration + ledger posting (5.3), package burn + ledger entries (5.4), consolidated group view (5.5), ledger↔chain reconcile-and-correct (5.6). This story delivers the **connection + watcher seams** those stories consume.
- OUT OF SCOPE (ops, deferred): the real Sepolia wiring/broadcast. Record it in `deferred-work.md` (it joins the 4.6 ops-deferred items). No real secret is ever created.

### ABI inference approach

Embed the ABIs as `as const` arrays in `src/abis/*.ts` (copied from `prod/contracts/out/**/<C>.json` `.abi`). viem requires the `as const` literal type to infer event arg / read return types. Do NOT import the full Foundry JSON at runtime (it is a build artifact outside the pnpm workspace and would couple TS build to `forge build`); a curated `as const` subset keeps the `chain` package self-contained and type-safe. Add a comment pointing back to the artifact so drift is auditable.
