# Deferred Work

## Deferred from: code review of story-3.3 (2026-06-16)

- **Bind authorization facts to persisted account state** — `postTransfer` takes `accountType`, `assetKind`, `classification`, and `destinationKind` as caller-supplied facts and does not cross-check them against the real `from`/`to` account rows. This is the documented Story-3.3 trust boundary; cross-asset transfers are already fail-safe (the ledger rejects the unbalanced entry and nothing persists). DB-backed resolution/validation of these facts belongs to **Story 3.4** (the generated off-chain policy provider over `flow_permissions` + `@rose/config` floor resolution).
- **Amount-sensitive authorization** — the `TransferScenario` carries no transaction `amount`, so per-transaction limits/velocity caps cannot be enforced at the chokepoint. By design: the scenario vocabulary is frozen in Story 3.1 and models floor-guarded flows abstractly via `ConformanceEnv.postBalanceBelowFloor` (computed at runtime in **Story 3.4**). Adding `amount` to the scenario would redefine the single-source rule vocabulary.
- **Aliased-import bypass of the static chokepoint guard** — the hardened `DIRECT_POSTINGS_INSERT` regex now catches bare, namespace-qualified (`insert(schema.postings)`), and quoted-identifier writes, but a fully aliased import (`import { postings as p }; db.insert(p)`) remains beyond a static regex. This is the accepted residual limit of a dependency-scan regression lock (AC-2 explicitly allows a "static/dependency check"); a future hardening could use an AST/semantic scan if warranted.

## Deferred from: code review of story-3.4 (2026-06-16)

- **Classification & destinationKind remain caller-asserted facts (the residual half of the 3.3 trust boundary)** — Story 3.4 binds the SOURCE `accountType` to the persisted `accounts` row (`assertAccountTypeMatches`, fail-closed). But `from.classification` (PRINCIPAL/YIELD/NONE) and `to.destinationKind` (TREASURY/CLIENT_ACCOUNT/EXTERNAL) are NOT persisted in P0 — the `accounts` row exposes only `type/asset/decimal_scale`, and there is no account→destinationKind mapping — so they cannot be cross-checked against persisted state and remain trusted caller inputs. They are load-bearing for the Model-A principal-egress prohibition and the destination match, so a caller could in principle mis-declare them. The **authoritative** off-chain enforcement of the principal/yield distinction is architecturally deferred to **Epic 4** (segregated principal sub-positions on-chain, FR-19); persisting a classification/destination-kind mapping off-chain (and validating it in `enforceTransfer`) is the follow-up. Documented as a code-comment trust boundary in `enforce-transfer.ts`.
- **Floor/scale parse errors surface as raw `RangeError`, not a modeled refusal** — when `BACKING_FLOAT_FLOOR` has more fractional digits than the account scale (or an account's `decimal_scale` is inconsistent with a known asset), `fromDecimalString` throws a `RangeError` inside `resolveOffChainEnv` before the chokepoint runs. This is fail-closed (no transfer persists), but the error escapes the `InvalidTransferError`/`TransferRefusedError` contract that an API boundary maps to 4xx, so it would read as a 500. Mapping config/parse faults to a typed refusal belongs at the **Epic 6** REST boundary. (A misconfigured NEGATIVE floor IS now handled in-story: it is treated as no usable floor ⇒ REFUSE, never a silent ALLOW.)

## Deferred from: code review of story-4.1 (2026-06-16)

- **Empty required-claim-topics ⇒ every registered holder is "verified" (ERC-3643 semantics)** — `IdentityRegistry.isVerified` returns `true` for a registered holder when `ClaimTopicsRegistry.getClaimTopics()` is empty (the per-topic loop is skipped). This matches the canonical ERC-3643 reference and is fail-safe in production because the single source `@rose/rule-spec` `eligibility.requiredClaimTopics` is `.min(1)` and **Story 4.5** seeds the on-chain topics FROM that rule-spec (so an empty set is only a transient admin/deploy window). Documented as a code comment; an explicit on-chain "require ≥1 topic" guard is deferred to **Story 4.5** (rule-spec → on-chain config derivation), which owns topic seeding.
- **Per-identity claims-per-topic is unbounded (self-griefing DoS only)** — `ClaimTopicsRegistry` caps the TOPIC axis (`MAX_CLAIM_TOPICS = 15`) and `TrustedIssuersRegistry` caps topics-per-issuer, but a holder who controls a CLAIM key on their own identity can add unboundedly many junk claims for a required topic (each `claimId = keccak256(issuer, topic)`, so distinct bogus issuer addresses yield distinct ids). `isVerified` loops those claims and could hit out-of-gas (try/catch does NOT catch OOG). Blast radius is the holder's OWN verification only (self-inflicted); matches ERC-3643. A per-identity/per-topic claim cap (or capping the verification scan) is deferred — revisit when `isVerified` is wired into the token transfer hot path in **Story 4.2**.
- **No EIP-1271 (contract-signer) claim issuers** — `ClaimIssuer.isClaimValid` validates only EOA ECDSA signatures (recovered CLAIM key) over an EIP-191 digest; a claim signed by a smart-contract signer can never validate (fail-closed, not a vulnerability). P0's claim-issuer operating model uses an EOA signing key, so this is acceptable now; EIP-1271 support is a future enhancement if a contract-based issuer signer is ever required.

## Deferred from: code review of story-4.2 (2026-06-16)

- **No forced-transfer / recovery and no pause / per-address freeze on `RoseToken`** — the token exposes only eligibility-gated `transfer`/`transferFrom`/`mint`/`burn`. There is no agent power to claw back or relocate a frozen holder's balance to a new wallet, and no emergency pause or address-level freeze. This is explicitly the scope of **Story 4.6** (gated transfer-agent powers: forced transfer, recovery, freeze, pause + Sepolia deploy). 4.2's `burn` exemption (issuer can reduce supply of a revoked holder) is the only supply-side lever shipped now; full recovery-to-new-wallet remains 4.6.
- **`RoseToken.setIdentityRegistry` has no holder-continuity check** — an owner swap to a registry that does not recognize the current holders (wrong/empty registry, different topics/issuers) freezes every holder's balance until the swap is reverted. Owner-trusted and recoverable (swap back), and mirrors stock ERC-3643 `setIdentityRegistry` which also has no continuity guard. A migration/continuity check (or a two-step registry swap) is an ops/4.6 hardening, not a 4.2 requirement.
- **`Ownable` single-key admin on `RoseToken`** — `mint`/`burn`/`setIdentityRegistry` are all gated by one `Ownable` owner. `renounceOwnership()` (inherited) would permanently brick those privileged ops; single-step `transferOwnership` is a fat-finger footgun; there is no minter/agent role separation or multisig. Consistent with the rest of the suite's `Ownable` usage; the transfer-agent role + multisig/role-separation lands with **Story 4.6** / deployment ops.
- **Unbounded claims-per-topic OOG remains relevant on the transfer hot path** — `RoseToken._update` calls `isVerified(from)`/`isVerified(to)` on transfers, which loop a holder's (uncapped — see story-4.1 deferral above) claims for each required topic; try/catch does NOT catch out-of-gas. After the 4.2 burn-exemption patch the burn path no longer loops the holder's claims, so the blast radius is back to **holder-self-griefing only** (a holder can make their OWN balance untransferable by bloating their claim list, but cannot block the issuer's burn). A per-identity/per-topic claim cap (or a bounded verification scan) should be revisited alongside **Story 4.6** recovery design.

## Deferred from: code review of story-4.4 (2026-06-16)

- **Burn retires principal with no leg-level protection beyond owner + coupling gating** — the Model-A bright line (`require(balanceOf(from) >= _principal[from])`) lives only in the `_update` transfer branch; the burn branch (`to == 0`) instead clamps `_principal[from]` down to the post-burn balance (`PrincipalReducedOnBurn`). So principal value is intentionally NOT protected from being burned — it is erased to stay consistent. This is by-design: a burn is `onlyOwner` (the `CoupledPair`) AND coupling-gated (`pairingInProgress`), so no holder can trigger it, and redemption is meant to retire the coupled package incl. principal. The authoritative recovery / forced-burn / "is this burn an authorized redemption" semantics belong to **Story 4.6** (gated transfer-agent powers) and the reset/settlement mechanics of **Epics 5-7**. (Raised by Blind Hunter, Med — design boundary.)
- **Additive `designatePrincipal` with no un-designate/release path → owner can permanently lock a holder's transferability** — `designatePrincipal` only ever increases `_principal[holder]` (`_principal + amount`), and the sole decrease path is the burn clamp; there is no `reducePrincipal`/`releasePrincipal`. An over-designation (owner error or compromised/buggy owner) bricks a holder's transfers until a pair-driven burn occurs, which a normal holder cannot trigger (single-leg burn is blocked by the coupling guard). Owner-trusted (the issuer) and recoverable via burn-and-remint at reset; a "release / reclassify / crystallize principal" path is **reset / P&L crystallization (D1a), Epics 5-7**, and the single-key `Ownable` DoS surface mirrors the existing 4.2/4.3 `Ownable` deferrals (multisig / role-separation lands with **Story 4.6**). (Raised by Blind Hunter, Med.)
- **Stranded-principal recovery for a revoked holder requires burning the counterparty's leg** — a revoked holder's principal can only be retired via `burnPair`, which forcibly burns an equal amount of the OTHER leg's holder; there is no single-leg principal release. By-design for 4.4 (the coupling primitive moves both legs); single-leg recovery / forced transfer to a new wallet is **Story 4.6**. (Raised by Edge-Case Hunter, Low.)
