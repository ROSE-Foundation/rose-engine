# Story 6.6: Exchange/Trading and Subscriber surfaces on live data

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Subscriber (Rose Member),
I want the Subscriber surfaces (and the Exchange/Trading view) functional on live data,
so that I can subscribe/redeem and view my position, and operators can see trading live (FR-14, UX-DR4, UX-DR6, UX-DR8).

## Acceptance Criteria

1. **Given** the live system (reusing the Story 6.5 design system + shared components), **when** I open the **Exchange/Trading view** (operator desktop), **then** it renders live (paper/testnet) execution data, positions, and P&L **by entity** (no mockups), using the `MoneyCell` / `DeltaIndicator` / `LiveIndicator` components and explicit **loading / empty / error** states (UX-DR1, UX-DR2, UX-DR4).
2. **Given** an eligible Subscriber on the **responsive** Subscriber surfaces (UX-DR8), **when** they use the surfaces, **then** the surfaces drive the live subscription/redemption flow (FR-11) via the **Review‚ÜíConfirm + pending** pattern (UX-DR6) and display the live position and embedded pair state, with explicit **loading / empty / error** states.
3. **Given** the Review‚ÜíConfirm write flow, **when** the Subscriber confirms a subscribe or redeem, **then** the UI stays **pending** ("Awaiting Sepolia confirmation‚Ä¶", `pending` badge) until the status endpoint flips `pending ‚Üí confirmed` ‚Äî **no optimistic success** (NFR-9, UX-DR6) ‚Äî and a typed refusal (DENY ‚áí 403, REFUSE ‚áí 422, conflict/idempotency ‚áí 409, refuse-if-absent ‚áí 503) is surfaced as an explicit message **naming the refusing rule** (the machine `code`), never a silent success or a generic block (NFR-4, UX-DR5).
4. **Given** the **eligibility gate** (UX-DR6, FR-19), **when** a Subscriber has no valid ONCHAINID eligibility claim, **then** the subscribe path is **unavailable with an explicit reason** ("Subscription unavailable ‚Äî eligibility claim not found. Contact your administrator.") ‚Äî no self-service KYC, not a generic block screen.
5. **Given** the responsive form-factor (UX-DR8), **when** the Subscriber surfaces are built, **then** they adapt **single-column (phone) ‚Üí centered reading column (`max-w-2xl`, desktop)**, the confirm panel is full-width on mobile, and the surfaces meet **WCAG 2.2 AA** in both color modes (no color-only signaling, full keyboard operability, money announced with unit + scale).
