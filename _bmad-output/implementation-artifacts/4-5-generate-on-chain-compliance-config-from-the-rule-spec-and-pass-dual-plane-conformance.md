# Story 4.5: Generate on-chain compliance config from the rule-spec and pass dual-plane conformance

Status: done

## Story

As a build engineer,
I want the on-chain compliance configuration emitted from the SAME `@rose/rule-spec` as the off-chain rules, and the shared conformance vectors run against an on-chain plane adapter,
so that the off-chain and on-chain authorization planes are provably derived from one source and cannot silently diverge (FR-19, §8 Q5, SM-4).

## Acceptance Criteria

**AC-1 — The on-chain compliance config is GENERATED (not hand-edited) from the single rule-spec, consistent with the off-chain `flow_permissions`**
**Given** the `@rose/rule-spec` codegen
**When** it emits the on-chain compliance configuration
**Then** the on-chain rules (required claim topics, allowlist requirement, coupling + Model-A params, allow flows, prohibitions, floor guards) are derived from `ruleSpecV1` — never hand-authored per-plane — and a drift guard fails on any hand-edit; the previously hand-pinned `ClaimTopics.ONCHAINID_KYC` (Story 4.1) is replaced by the GENERATED value and the `ClaimTopicsRegistry` is seeded from the generated topics; the deferred "≥1 required topic" hardening (signalled by 4.1/4.2) lands here as a fail-closed guard.

**AC-2 — The shared conformance vectors pass against the on-chain plane with the SAME decisions as the off-chain plane (dual-plane equivalence)**
**Given** the shared `conformanceVectors` (10 vectors, tagged for BOTH planes) and the reusable harness (`runConformance`/`assertAllConform`, `PlaneAdapter`)
**When** they are executed against an ON_CHAIN `PlaneAdapter` derived from the generated on-chain config
**Then** every vector passes with the SAME allow/deny/refuse outcome as the off-chain plane (SM-4 dual-plane equivalence), and the convened equivalences are encoded and proven on the real contracts: curated allowlist == `IdentityRegistry`, eligibility topic == `uint256(keccak256("ONCHAINID_KYC"))`.

### Single-source / dual-plane architecture (the heart of 4.5)

- **`@rose/rule-spec` `ruleSpecV1` is THE source of truth.** The off-chain plane derives `generateOffChainPolicy(ruleSpecV1)` → `flow_permissions` (3.4). 4.5 adds the SYMMETRIC on-chain derivation `generateOnChainComplianceConfig(ruleSpecV1)` → on-chain compliance config (claim topics, eligibility, coupling, Model-A, flows). Edit rules ONLY in `rule-spec.v1.ts`; both generated artifacts are byte-identical-on-regen and drift-guarded (the hand-edit tripwire). [Source: rule-spec.v1.ts:1-6; generate-off-chain-policy.ts:1-7; codegen/cli.ts:1-6]
- **Reuse the conformance harness AS-IS.** `runConformance(adapter, vectors)` filters by `adapter.plane`; the shared `conformanceVectors` are tagged for BOTH planes. The on-chain adapter is the FINAL plane the harness was designed for — `types.ts` and `harness.ts` literally say "Epic 4 supplies the on-chain adapter … the SAME vectors run against all planes so the two rule sets cannot silently diverge." Do NOT add an on-chain-only vector; run the existing 10. [Source: conformance/harness.ts:1-21; conformance/types.ts:1-14; conformance/vectors.ts:1-13]
- **The on-chain reference adapter mirrors the off-chain reference adapter's fail-closed resolution.** They MUST agree on every vector — that agreement IS the SM-4 proof. Keep the resolution order identical (prohibition→allow-match→floor→default-deny); the only difference is the artifact TYPE it reads (`OnChainComplianceConfig` vs `OffChainPolicyArtifact`). If they ever disagree, the cross-plane equivalence test (Task 5c) fails — exactly the divergence guard FR-19 demands. [Source: conformance/reference-off-chain-adapter.ts:34-101]
- **Convened on-chain equivalences (encode in adapter NatSpec; PROVE on real contracts in Task 8):** curated allowlist == `IdentityRegistry` (registration presupposes off-chain KYC/AML); eligibility topic == `uint256(keccak256("ONCHAINID_KYC"))`; Model-A bright line == the on-chain segregated-principal guard (4.4); pair coupling == `CoupledPair` atomic mint/burn (4.3). The 10 flow vectors are evaluated by the TS adapter; the eligibility/topic/seeding/coupling/Model-A primitives are proven on the actual EVM contracts via `forge test`. [Source: epics.md#Story 4.5 AC; IdentityRegistry.sol:12-17; ClaimTopics.sol:4-17]

### Why no keccak256 in TS (dependency discipline)

- The canonical on-chain topic id is `uint256(keccak256(utf8(label)))` — Node's `crypto` exposes SHA3-256 (FIPS padding), NOT Ethereum keccak, so it cannot reproduce the EVM value; adding a keccak dependency would violate the "no unneeded deps / zod-only-leaf" posture. Instead the TS config carries the topic LABEL (the single-sourced input) and the GENERATED Solidity materializes the numeric id where the EVM is authoritative. A Vitest test proves the JSON label == spec label == the literal embedded in the generated `.sol`; a forge test proves `keccak256(label)` (computed by the EVM) == `ClaimTopics.ONCHAINID_KYC`. Together they tie label→id across both planes with NO TS hashing. [Source: rule-spec/package.json deps (zod only); rule-spec-schema.ts:6-15 "schema is a leaf: imports only zod"; CLAUDE.md dependency rule]
